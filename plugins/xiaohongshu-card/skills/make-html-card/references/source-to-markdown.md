# Source → Markdown

无论输入是什么，Phase 1 都先转成统一的 `source/source.md`，并把不确定项写进
`source/extraction-notes.md`。后续 Phase 2+ **只面对这一份文件**，不再触碰原始输入。

## 产物

`source/source.md` 要包含：

- **标题** · **来源**（URL / 文件名 / "用户粘贴"）
- **正文**：保留段落结构
- **结构信号**（对 Phase 2 拆卡至关重要，必须保留）：
  - 段落锚点：emoji 前缀（👉 / 📌 / 💡 / ✅ 等）、markdown `##` / `###` 标题、【】方括号标题
  - 列表：`•` / `-` / `1.` 等格式原样保留
  - 核心术语 / 关键数据 / 需强调词组（可随手用 `**加粗**` 标出，Phase 2 会据此做 `<span class="highlight">`）
  - 可能的封面主题词（2-4 个字概括全文）
- **配图提示**：原文提到的图 / 表，用占位标注（`![占位：xxx](placeholder)`），等用户补图

`source/extraction-notes.md` 要记录：

- 输入类型 · 提取方式
- 可能丢失的信息（PDF 表格、网页图片、脚注）
- 拿不准的层级 / 术语
- 需要用户补充的素材（如封面配图）

原件留底到 `source/original.*`（URL 抓取的 HTML、PDF、原文本都存一份，供 diff 核查）。

## 不同输入的处理

| 输入 | 处理方式 | 自检重点 |
|---|---|---|
| 纯文本 / 用户粘贴文案 | 直接识别结构信号并标注；**不擅自编造层级** | emoji 锚点 / 列表格式是否保留 |
| 网页 URL（公众号 / 竞品小红书 / 文章） | 抓正文，清理导航 / 广告 / 推荐 / Cookie 横幅 | 是否抓到主体，结构是否塌成一片 |
| PDF | 提取文本，尽量保留标题层级；表格 / 图用占位 | 断行错乱、页眉页脚混入、两栏阅读顺序 |
| Markdown | 保持原有标题 / 列表 / 加粗 | 不要过度改写源文 |

URL 抓取：优先用 agent 的网页抓取能力（或 `/browse`）获取正文，再清理噪音；抓完存一份 `source/original.html` 留底。

## Source Phase 自检（主 Agent 内联 · 5 条）

源材料质检**不是硬性 SubAgent 质检点**。主 Agent 进 Phase 2 前反正要通读 `source.md`，
就地按这 5 条核查、按结论修复即可（无需单独的 review 文件）：

1. **完整性**：是否被截断？长文 / 长 PDF 是否只抓到首屏或一半？
2. **结构信号**：Phase 2 要用的锚点（emoji / ## / 【】/ 列表）是否保留？还是被压平成一片正文，后面没法分卡？
3. **关键载体**：表格 / 代码 / 引用 / 数据是否保留且没损坏（表格没挤成一行、数据没丢）？
4. **噪声**：是否误把导航 / 广告 / 推荐阅读 / 页眉页脚写进正文？编码有没有伤（乱码、软连字符断词）？
5. **封面线索**：能否从 source.md 看出 2-4 字主题词？拿不准的都写进 `extraction-notes.md` 了吗？

> ⚠️ 第 1 / 4 条光看 markdown 就能抓到，但 **2 / 3 里"静默丢失"的锚点 / 表格**（被悄悄吞掉）
> 光看 source.md 看不出来——必须对照 `original.*` 做 diff。
